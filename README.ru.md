# LeadSpawn — личный разбор проекта с нуля

Это подробное объяснение проекта LeadSpawn на русском языке: что это, из чего
состоит, как работает каждая часть и почему приняты те или иные технические
решения. Документ рассчитан на то, чтобы понять проект «с нуля», даже если ты
подзабыл детали.

---

## 1. Что такое LeadSpawn

LeadSpawn помогает фрилансерам и агентствам находить потенциальных клиентов.
Пользователь вводит:

- **нишу** (тип бизнеса, например «dentist»),
- **страну**,
- **город**,

а приложение возвращает список компаний с названием, адресом, телефоном,
сайтом и координатами. Результат можно посмотреть в виде карточек или таблицы
и выгрузить в CSV.

Данные берутся из **OpenStreetMap** — это бесплатно и не требует API-ключа.

---

## 2. Общая архитектура

Поток данных сверху вниз:

```
Браузер (React SPA)
      │  POST /api/v1/search  { query, city, country, limit, has_website, has_phone }
      ▼
FastAPI (app/api/search.py)
      │
      ▼
Сервис-оркестратор (app/services/leads.py)
      │
      ├─► Есть свежий кэш в PostgreSQL? ──да──► отдать из базы (миллисекунды)
      │
      └─► нет ──► app/services/overpass.py
                     │
                     ├─ Nominatim: город+страна → координаты (bounding box)
                     └─ Overpass: найти бизнесы внутри этого прямоугольника
                     │
                     ▼
                  сохранить в PostgreSQL и вернуть
```

Ключевая идея: приложение состоит из двух отдельных частей — **бэкенд** (Python
API) и **фронтенд** (React-приложение). Они общаются по HTTP через один эндпоинт
`POST /api/v1/search`.

## 3. Технологический стек и зачем каждый инструмент

**Бэкенд:**

- **Python 3.12** — язык.
- **FastAPI** — веб-фреймворк. Быстрый, асинхронный, сам генерирует
  документацию (`/docs`) из типов.
- **Pydantic** — валидация входных данных и описание схем ответа.
- **httpx** — асинхронный HTTP-клиент для запросов к OpenStreetMap.
- **SQLAlchemy (async)** — ORM: описываем таблицы Python-классами, а не SQL.
- **asyncpg** — асинхронный драйвер PostgreSQL.
- **Alembic** — миграции базы данных (версионирование схемы).
- **PostgreSQL** — база данных, в которой хранится кэш поисков.

**Фронтенд:**

- **React 19 + TypeScript** — UI и типобезопасность.
- **Vite** — сборщик и dev-сервер.
- **Tailwind CSS v4** — стили через утилитарные классы.
- **shadcn/ui-подобные компоненты** — свои маленькие UI-компоненты (кнопка,
  карточка, таблица и т.д.) в `frontend/src/components/ui/`.
- **Framer Motion** — анимации.
- **react-i18next** — переводы (EN/RU/DE/ES/TR).
- **sonner** — всплывающие уведомления (тосты).

**Инфраструктура:**

- **Docker + docker-compose** — упаковка всего стека (база + API + фронтенд).

## 4. Как устроен бэкенд

### 4.1. Точка входа и эндпоинт

Приложение создаётся в `app/main.py`: там подключается CORS (чтобы фронтенд с
другого порта мог обращаться к API), роутер поиска и обработчик жизненного
цикла (закрывает пул соединений с базой при остановке).

Сам эндпоинт — `app/api/search.py`. Он принимает тело запроса, отдаёт его
сервису и превращает исключения в понятные HTTP-коды:

- `404` — город не найден (`LocationNotFoundError`);
- `502` — источник данных недоступен (`LeadProviderError`);
- `422` — некорректные данные формы (это FastAPI/Pydantic делают сами).

### 4.2. Валидация через Pydantic

`app/schemas/search.py` описывает форму запроса `SearchRequest` и ответа
`SearchResponse`. Например, `query` должен быть от 2 до 80 символов и содержать
только безопасные символы (`pattern`), `limit` — от 1 до 50. Если данные не
проходят проверку, FastAPI автоматически вернёт `422` с описанием ошибки — нам
не нужно писать проверки вручную.

### 4.3. Откуда берутся данные: OpenStreetMap

OSM не умеет искать по фразе «дантисты в Берлине». Поэтому поиск идёт в два шага
(`app/services/overpass.py`):

**Шаг 1. Геокодинг (Nominatim).** Строку «город, страна» превращаем в
**bounding box** — прямоугольник координат `(юг, запад, север, восток)`.

> Почему bounding box, а не границы города? Раньше мы фильтровали по
> административной границе города (area). Для больших городов под нагрузкой
> публичные серверы Overpass не успевали и отдавали таймаут. Bounding box
> использует пространственный индекс напрямую и работает в разы быстрее
> (2–6 секунд вместо таймаутов на 30+ секунд). Расплата — прямоугольник немного
> выходит за границы города, но для лидов «соседний бизнес» — это нормально.

**Шаг 2. Поиск бизнесов (Overpass).** Запрашиваем именованные объекты внутри
прямоугольника, у которых теги соответствуют нише.

> Почему теги, а не поиск по названию? Сканировать регулярным выражением
> названия всех объектов большого города — слишком медленно, сервер отваливается
> по таймауту. Поэтому в `app/services/osm_tags.py` есть словарь `TAG_MAPPINGS`,
> который переводит понятные слова в точные теги OSM: `dentist →
> amenity=dentist`, `bakery → shop=bakery` и т.д. Точное совпадение по тегу
> быстрое и точное. Если слово незнакомо, включается «запасной» режим — поиск по
> набору тег-ключей регулярным выражением.

**Надёжность.** Публичные серверы Overpass часто перегружены, поэтому мы
перебираем несколько зеркал по очереди (`OVERPASS_URLS`). Есть тонкость: Overpass
на таймаут отвечает кодом **200**, но кладёт поле `remark` и пустой список. Если
бы мы этого не обрабатывали, пользователь увидел бы «ничего не найдено» вместо
ошибки. Поэтому пустой ответ с `remark` мы считаем ошибкой.

**Очистка результатов** (`_parse_and_dedup`): выбрасываем объекты без названия,
удаляем дубликаты (одинаковое имя в пределах ~100 м — так OSM хранит один и тот
же бизнес и как точку, и как здание), нормализуем пустые значения в `null`, к
голым сайтам вида `www.foo.de` добавляем `https://`.

### 4.4. Кэширование в PostgreSQL

Запрос к Overpass медленный (5–35 секунд). Чтобы не ходить туда повторно, мы
кэшируем результат в базе (`app/services/leads.py`).

Логика такая:

1. Нормализуем ключ: `(query, city, country)` приводим к нижнему регистру.
2. Ищем в базе свежую запись (не старше `CACHE_TTL_DAYS`, по умолчанию 30 дней).
3. **Есть кэш** → достаём бизнесы из базы, применяем фильтры и лимит, отдаём с
   флагом `cached: true`. Это занимает миллисекунды.
4. **Нет кэша** → идём в Overpass, сохраняем **полный** набор (до 150 бизнесов),
   потом применяем фильтры/лимит.

> Важное решение: кэш хранит **полный, нефильтрованный** набор на ключ
> `(query, city, country)`. А `limit`, `has_website`, `has_phone` применяются
> уже поверх кэша. Благодаря этому одна запись кэша обслуживает много разных
> запросов: «dentist Berlin 50 штук», потом «dentist Berlin 5 штук только с
> телефоном» — второй запрос не идёт в Overpass вообще.

Замер: первый запрос ~35 секунд, повторный ~0.23 секунды — ускорение примерно
в 150 раз.

### 4.5. Конфигурация

`app/core/config.py` — класс `Settings` на базе `pydantic-settings`. Он читает
значения из файла `.env` и переменных окружения. Так секреты (пароль базы,
ключи API) не попадают в код. Пример переменных — в `.env.example`.

## 5. База данных

Две таблицы (`app/models/search.py`):

- **`cached_searches`** — одна строка на поиск: нормализованные `query`, `city`,
  `country`, количество результатов, время загрузки. Есть уникальное ограничение
  на тройку `(query, city, country)`, чтобы не было дублей.
- **`cached_leads`** — строки-бизнесы, привязанные к поиску через `search_id`
  (при удалении поиска удаляются каскадно).

Схема создаётся не «на лету», а через **миграции Alembic** (папка
`alembic/versions/`). Команда `alembic upgrade head` создаёт таблицы с нуля. Если
поменяешь модели — сгенерируй новую миграцию:
`alembic revision --autogenerate -m "описание"` и снова `alembic upgrade head`.

## 6. Как устроен фронтенд

### 6.1. Структура

```
frontend/src/
├── components/
│   ├── Hero, Particles, SearchForm, Header, Footer
│   ├── ThemeProvider, ThemeToggle, LanguageMenu
│   ├── results/  (ResultsSection, LeadCard, LeadTable, LeadActions, ResultStates)
│   └── ui/       (button, card, input, table, switch, dropdown-menu, ...)
├── hooks/useLeadSearch.ts   — главная логика поиска
├── lib/ (api.ts, export.ts, utils.ts)
└── i18n/locales/ (en, ru, de, es, tr)
```

### 6.2. «Живое» появление результатов

Бэкенд возвращает все результаты одним ответом. Но чтобы поиск ощущался «живым»,
хук `useLeadSearch.ts` не показывает всё сразу: результаты хранятся в очереди и
появляются по одному с анимацией, а счётчик найденных растёт на глазах. Это
клиентская иллюзия «обнаружения в реальном времени» — без усложнения бэкенда.

Хук — это маленький конечный автомат со статусами:
`idle → searching → revealing → done | error`.

### 6.3. Тема, языки, экспорт

- **Тема** (`ThemeProvider.tsx`): светлая/тёмная, переключение анимировано,
  выбор сохраняется в `localStorage`, при первом заходе учитывается системная.
- **Языки** (`i18n/`): пять языков, переключаются через иконку глобуса. Добавить
  новый язык = один JSON-файл в `i18n/locales/` + одна строка в `LANGUAGES`.
- **Экспорт CSV** (`lib/export.ts`): экспортёры лежат в списке `exporters`, так
  что позже легко добавить, например, Google Sheets, не трогая интерфейс.

> Почему CSV сделан через разделитель `;` и строку `sep=;`? Обычный CSV с
> запятыми открывался в Excel одной колонкой. Причина не в баге, а в локали:
> Excel во многих европейских (и не только) локалях ожидает `;` как разделитель
> списка. Директива `sep=;` в первой строке заставляет Excel разбить файл на
> колонки независимо от локали. В начале файла также стоит BOM, чтобы Excel
> корректно читал UTF-8 (буквы с умлаутами, кириллицу и т.п.).

## 7. Docker и docker-compose

Весь стек можно поднять одной командой. Три сервиса (`docker-compose.yml`):

- **db** — PostgreSQL 17 (данные хранятся в томе `pgdata`, есть healthcheck).
- **api** — образ из корневого `Dockerfile`: ставит зависимости, при старте
  выполняет `alembic upgrade head` и запускает uvicorn. Ждёт, пока база станет
  «здоровой».
- **web** — образ из `frontend/Dockerfile`: собирает фронтенд и отдаёт его через
  nginx. Nginx же проксирует запросы `/api/…` на сервис `api` (см.
  `frontend/nginx.conf`), поэтому фронтенд и API для браузера — один origin, и
  CORS не нужен.

Запуск:

```bash
docker compose up --build
```

Открыть: http://localhost:8080 (фронтенд), http://localhost:8000/docs (API).

## 8. Как запустить локально (без Docker)

Нужен установленный и запущенный PostgreSQL на порту 5432 и созданная база
`leadspawn`.

Бэкенд (из корня репозитория):

```powershell
python -m venv .venv
.venv\Scripts\pip install -r requirements.txt
copy .env.example .env
.venv\Scripts\alembic upgrade head
.venv\Scripts\uvicorn app.main:app --reload
```

Фронтенд (нужен Node 20+; pnpm через corepack):

```powershell
cd frontend
corepack pnpm install
corepack pnpm dev
```

Открыть http://localhost:5173. Dev-сервер сам проксирует `/api` на порт 8000.

## 9. Частые вопросы

- **Первый поиск долгий, повторный мгновенный.** Так и задумано: первый идёт в
  OpenStreetMap, повторный берётся из кэша в PostgreSQL (бейдж «Instant / из
  кэша»).
- **По какому-то запросу ничего не находится.** Возможно, нет соответствия в
  `TAG_MAPPINGS`. Добавь нишу в `app/services/osm_tags.py`.
- **Некоторые бизнесы без телефона/сайта.** Данные OSM заполняются сообществом,
  полнота зависит от города. Для этого и есть фильтры «есть телефон / есть сайт».

## 10. Как расширять

- **Новая ниша:** добавь запись в `TAG_MAPPINGS` (`app/services/osm_tags.py`).
- **Новый язык:** JSON-файл в `frontend/src/i18n/locales/` + строка в `LANGUAGES`.
- **Новый формат экспорта:** добавь объект в `exporters` (`frontend/src/lib/export.ts`).
- **Переход на Google Places** (платно, но данные полнее): в
  `app/services/google_places.py` уже есть клиент; нужно подставить его вместо
  Overpass в слое сервисов и задать `GOOGLE_PLACES_API_KEY` в `.env`.
